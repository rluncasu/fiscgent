### Planner Response

Great question. Traditional chunking is indeed problematic for legal documents — you risk splitting articles mid-sentence, losing cross-references, and destroying the hierarchical context that gives legal text its meaning. Here are the viable alternatives:

---

### The Problem with Chunking

The Cod Fiscal is structured as: **Titlu → Capitol → Secțiune → Articol → Alineat** + linked **Norme Metodologice** per article. An Article is the **atomic legal unit** — it's self-contained and internally coherent. Arbitrary 512-token chunks would:
- Split an Article across multiple chunks, losing its meaning
- Separate an Article from its associated Norme
- Lose the hierarchical path (which Titlu/Capitol this belongs to)
- Break cross-references ("conform art. 7 alin. (1) pct. 21")

---

### Alternative Strategies

| Approach | How it works | Pros | Cons |
|---|---|---|---|
| **1. Article-as-document** | Parse Cod Fiscal into its natural legal units (Articles). Each Article = one indexable document, stored whole with its Norme | Preserves legal integrity; no meaning loss | Some articles are very long, some very short — uneven retrieval |
| **2. Structured SQLite + FTS5 + Vector** | Relational DB with tables for `titluri`, `capitole`, `articole`, `norme`. FTS5 keyword search + vector embeddings on each row | 3 retrieval modes (exact lookup, keyword, semantic); aligns with Hermes internals (already uses SQLite/FTS5) | More complex to build |
| **3. Parent-document retriever** | Embed smaller sub-sections for matching, but at retrieval time return the **full parent Article + Norme** | Best of both worlds: fine-grained matching, full-context delivery | Requires two-tier storage |
| **4. Full-context with selective retrieval** | Store entire Titles as documents; retrieve relevant Title then pass whole Title to LLM | Maximum context preservation | Titles can be 50+ pages; wastes tokens |

---

### My Recommendation: **Option 2 — Structured SQLite + FTS5 + Vector Hybrid**

This is the strongest fit for several reasons:

1. **Three retrieval paths in one system**:
   - **Exact lookup**: "Ce spune Art. 291?" → direct SQL `WHERE articol = '291'`
   - **Keyword search**: "TVA intracomunitar" → FTS5 full-text search
   - **Semantic search**: "ce taxe plătesc pe dividende?" → vector similarity

2. **Preserves the legal hierarchy completely**:
   ```
   articole table:
   ├── id, titlu_id, capitol_id, sectiune_id
   ├── articol_number ("291")
   ├── articol_title ("Cotele de TVA")
   ├── full_text (complete article, unmodified)
   ├── norme_text (associated methodological norms)
   ├── section_path ("Titlul VII > Cap. X > Secțiunea 1")
   ├── embedding (vector, for semantic search)
   └── cross_references ["art. 7", "art. 292"] (extracted)
   ```

3. **Architecturally consistent with Hermes** — Hermes already uses SQLite/FTS5 for session storage and memory. No new infrastructure needed.

4. **Retrieval returns the full Article + Norme** — never a fragment. Legal meaning is always preserved.

5. **Cross-reference resolution**: When Art. 291 references Art. 7, we can fetch both and include them in context.

---

- Use `sqlite-vec` extension (or inline embeddings) for vector search alongside FTS5

MemPalace is a fascinating option. Here's my honest assessment:

---

### What MemPalace does well

The **Palace metaphor maps beautifully to the Cod Fiscal structure**:

| Cod Fiscal Hierarchy | MemPalace Concept |
|---|---|
| Titlu (Title) — e.g., "Titlul VII - TVA" | **Wing** |
| Capitol (Chapter) | **Room** |
| Articol (Article) | **Drawer** (verbatim, never summarized) |
| Cross-references between articles | **Tunnels** (automatic cross-wing links) |
| Norme Metodologice per article | **Closet** (summary pointing to drawer) |

Key advantages over chunking:
- **Raw verbatim storage** — stores full articles as-is, no lossy chunking
- **Structured navigation** — Wing → Room → Drawer gives the agent a map, not just flat search
- **Tunnels** — automatic cross-references when the same topic appears in different Titles
- **96.6% recall on raw mode** — proven retrieval quality
- **ChromaDB-backed + local** — no external APIs needed
- **Has an MCP server** with 29 tools — could connect directly to Hermes via MCP integration
- **Knowledge Graph** — can model relationships like "Art. 291 references Art. 7"

### Concerns

1. **Designed for conversations, not legal documents** — The `mine` command and miners expect chat exports / project files. We'd need to use the **Python API directly** (`add_drawer`) to ingest structured legal text, not the CLI miners.

2. **Very young project** (v3.1.0, April 2026) — Acknowledges bugs: macOS ARM64 segfault (#74), ChromaDB pinning (#100), shell injection in hooks (#110). The team is honest about issues but it's still early.

3. **AAAK is not useful here** — The compression dialect is irrelevant for legal text where precision matters. We'd use raw mode only.
